#  TraceDetectiveAndroid

> **"Data doesn't lie. The culprit is within the trace."**

`TraceDetectiveAndroid`는 안드로이드 시스템 트레이스(perfetto)를 분석하여
성능 저하(Regression)의 근본 원인을 추적하는 **AI 기반 성능 포렌식 엔진**입니다.
Gemma 3 12B 모델과 8개의 정예 수사 도구(API)를 결합하여,
단순한 "느림"을 넘어 **"왜, 어디서, 누구 때문에" 느려졌는지**에 대한 인과관계를 입증합니다.

---

##  Purpose (수사 목적)

안드로이드 성능 분석은 수만 개의 슬라이스와 복잡한 스케줄링 데이터 사이에서
'범인'을 찾는 고통스러운 과정입니다. 본 프로젝트는 다음을 목표로 합니다:

1. **Delta (Δ) 기반 분석**
   - 정상(Normal)과 지연(Slow) 트레이스를 대조하여 퇴보 지점을 정밀 타격
2. **인과관계 규명**
   - 메인 스레드 지연이 Binder 통신 때문인지, Lock 경합 때문인지,
     혹은 외부 프로세스의 CPU 가로채기(Steal) 때문인지 밝힘
3. **자율 수사**
   - AI 수사관이 가설을 세우고 적절한 도구를 호출하여 스스로 최종 보고서를 작성

---

##  Strategy: The Precision Δ Method

우리는 수치로만 말합니다. 모든 분석은 다음 공식을 따릅니다:

```math
\Delta T = T_{slow} - T_{normal}
```

---

##  Forensic Tools (The Magnificent Seven + 1)

수사관은 현장을 완벽하게 통제하기 위해 8개의 전용 API를 사용합니다:

- `initial_system_scan`  [초동 수사] 마일스톤 지연 파악 및 전체 시스템 CPU 점유 Top15 식별
- `profile_main_thread`  UI 스레드 내 함수별 Self-time 및 Total-time 분석
- `trace_binder_calls`  IPC 지연 및 응답 지연을 유발한 상대방(Destination) 프로세스 추적
- `check_thread_states`  CPU 스케줄링 상태(Running, Runnable, Sleep) 분석
- `check_lock_contention`  Java/Native Lock 경합 및 자원을 쥐고 있는 Owner 스레드 식별
- `analyze_memory_gc`  GC 발생 시점 및 UI 스레드 중단 시간 확인
- `execute_custom_sql`  특수 상황 분석을 위한 자유 SQL 쿼리 발부
- `` *(필요 시 추가 도구 확장 가능)*

##  Token Dashboard

수사관의 자원 소모량을 실시간으로 모니터링합니다. 효율적인 SQL 쿼리 설계로
최소한의 토큰으로 최대의 증거를 확보합니다.

```
Tokens: 12,450 / 32,768 (80%)
``` 
- **Input**: 수사 기록 및 증거 데이터 누적량
- **Output**: 수사관의 추론 및 지시 사항
- **Usage**: Gemma312B 컨텍스트 윈도우 점유율
